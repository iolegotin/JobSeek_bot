

# TECH SPECIFICATION: JobSeeker AI Bot (MVP) - V2

## 1. АРХИТЕКТУРА БАЗЫ ДАННЫХ
Проектирование схемы данных с использованием **PostgreSQL** и **SQLAlchemy 2.0 (Async)**. Ключевым элементом является использование типа `JSONB` для хранения динамических весов фильтров, что позволяет гибко адаптировать систему под пользователя без изменения структуры таблиц.

| Таблица | Описание | Ключевые поля / Структура |
| :--- | :--- | :--- |
| **users** | Кабинеты соискателей и администраторов. | `id` (UUID PK), `tg_chat_id` (BIGINT, уникальная связь), `role` (ENUM: 'user', 'admin'), `is_active` (BOOLEAN), `created_at` (TIMESTAMP). |
| **user_filters** | Настройки поиска и весов на естественном языке. | `id` (UUID PK), `user_id` (FK -> users), `query_text` (TEXT, естественный язык), `weights_config` (**JSONB**: хэш слов-ключей с числовыми весами `{'keyword': 1.0}`), `cooldown_interval` (INT, сек). |
| **vacancies** | Агрегированные предложения (HH.ru + TG-каналы). | `id` (UUID PK), `source_type` (ENUM: 'hh', 'tg_channel'), `external_id` (TEXT), `title` (TEXT), `raw_data` (**JSONB**, полный сырой ответ API/парсера), `is_processed_by_ai` (BOOLEAN, для контроля статусов). |
| **match_logs** | Логи матчинга и истории Feedback Loop. | `id` (UUID PK), `user_id` (FK), `vacancy_id` (FK), `ai_score` (FLOAT), `status` (ENUM: 'delivered', 'disliked', 'ignored'), `feedback_type` (ENUM: 'like', 'dislike'). |

**Связи и Индексация:**
*   Все FK-связи реализуются через `on_delete='CASCADE'`.
*   Индексы создаются на `(user_id, created_at)` для быстрой выборки истории.
*   JSONB индексы (`GIN`) создаются на полях весов для возможности поиска по ключам внутри фильтров при необходимости.

## 2. МОДУЛЬ ПАРСИНГА И АГРЕГАЦИИ
Асинхронный модуль сбора данных, строго соблюдающий лимиты внешних систем и обеспечивающий отказоустойчивую инкрементальную доставку.

### 2.1. Клиент `api.hh.ru` (Async Wrapper)
*   **Rate Limiting:** Реализация через `aiogram.DispatcherMiddleware` или отдельный менеджер. Перед отправкой запроса проверяется счётчик в Redis (`HINCRBY`).
*   **Обработка 429:** При получении `HTTP 429 Too Many Requests` система включает **экспоненциальную задержку** (Exponential Backoff: $T = T_{base} \cdot 2^n$). Парсер замораживается, отправляется алерт админу в TG-чат. Данные не отбрасываются, а кэшируются локально для повторной обработки при восстановлении.
*   **Параллельность:** Используются `asyncio.gather` с лимитом concurrency (например, 5 потоков), чтобы не перегрузить сеть и не вызвать бан.

### 2.2. Юзер-бот для TG-каналов (Pyrogram / Telethon)
*   **Инкрементальный захват:** Бот хранит `last_offset` или `max_id` последнего успешного сообщения из каждого целевого канала. Запросы кетают данные начиная от сохраненного маркера.
*   **FloodWait Handling:** Обработчик ошибок перехватывает `FloodWaitError`. При возникновении бота ставится на случайную задержку (Jitter 2–5 сек) между запросами к каналам, пока не накопится пакет данных или не истечет лимит.
*   **Склейка дубликатов:** Перед сохранением в БД вычисляется `hash(text_post + channel_id)`. Если хэш есть в кэше Redis (`EXPIRE 7 days`), вакансия игнорируется.

## 3. AI-СЛОЙ И FEEDBACK LOOP
Использование **LiteLLM** для унификации вызовов (поддержка локальных моделей `Ollama`, облачных `GPT`/`Claude`).

### 3.1. Структура Системного Промпта
Промпт динамически собирается из `query_text` и весов пользователя:
> *"Ты — эксперт по подбору вакансий. Профиль пользователя на естественном языке: {query_text}. Веса ключевых слов в JSONB: {weights_config}. Твоя задача — проанализировать вакансию, вернуть релевантный скор 0.0-1.0 и причины."*

### 3.2. Формат JSON-ответа от модели
Модель должна возвращать строго форматированный JSON:
```json
{
  "score": float (0.0 - 1.0), // Насколько вакансия подходит по текущим весам
  "reasons": ["Совпадает с зарплатой", "Нет опыта в required skills"],
  "updated_weights_suggestion": { // Опционально: для обучения на лету
    "python": +0.5, 
    "docker": -0.2
  }
}
```

### 3.3. Алгоритм динамической корректировки весов (Feedback Loop)
1.  **LIKE:** Если юзер жмет `+`, система берет текущие веса фильтра и **прибавляет** веса к словам, которые AI посчитал релевантными. Вес слова в JSONB увеличивается на `+0.5`.
2.  **DISLIKE:** При `-` веса анти-слов (слова из текста вакансии) увеличиваются по модулю, а положительные — обрезаются или прибираются. 
3.  **Адаптация:** Система не делает дообучения модели (fine-tuning). Она просто пересчитывает `weights_config` в JSONB для конкретного юзера, адаптируя "фильтр" под его реакции на лету.

## 4. СХЕМА ОЧЕРЕДЕЙ И БУФЕРИЗАЦИИ
Система построена на принципах буферизации и отказоустойчивости при сбоях нейросетей и сетей.

### 4.1. Очередь Rety LiteLLM (Redis)
*   **Канал `ai_analysis_queue`:** При сбое сети или недоступности LLM вакансия помещается в эту очередь с полем `retry_count`. 
*   **Worker:** Отдельный асинхронный воркер постоянно слушает эту очередь. Как только LiteLLM становится доступным, он подхватывает задачи из очереди и доанализирует пропущенные вакансии.

### 4.2. Буферизация рассылок (Notification Batching)
*   **Структура буфера в Redis:** `KEY = user_id:cooldown_bucket`. Храним массив ID вакансий, которые прошли фильтр.
*   **Batching Logic:** Сообщение отправляется не мгновенно, а накапливается в пакет. Через интервал `cooldown_interval` (например, 30 мин) запускается таймер. Если накопилось >1 вакансии, они группируются в один пост-кабелю ("Ваша подборка за период").
*   **Graceful Degradation:** При падении инфраструктуры или критических банах бот ставится на паузу (`STOP`), админу улетает алерт. Активные сессии юзеров получают вежливый системный статус: *"За последние 24 часа новых вакансий не найдено, я продолжаю поиск"*.

## 5. СТРУКТУРА ПРОЕКТА
Файловая система будущего приложения (`JobSeekerBot`):

```text
jobseeker_bot/
├── .env                    # Переменные окружения (API keys, DB creds)
├── pyproject.toml        # Зависимости: aiogram>=3.0, telethon/pyrogram, asyncpg, sqlalchemy[async], litellm
├── app/
│   ├── __init__.py
│   ├── main.py           # Точка входа asyncio.run(dispatcher)
│   ├── config.py         # Настройки констант
│   ├── router.py         # Маршрутизация (вешение хендлеров aiogram v3.x)
│   ├── middleware/       # Мидлвары для авторизации и логирования
│   └── handlers/         # Обработчики команд /!start, админ-панели
├── core/
│   ├── db.py             # Подключение SQLAlchemy 2.0 Async + Alembic миграции
│   ├── models.py         # ORM модели таблиц (users, filters, vacancies)
│   └── services/
│       ├── hh_parser.py    # Асинхронный клиент HH.ru с обработкой 429
│       ├── tg_fetcher.py   # Логика юзер-бота для инкрементального парсинга каналов
│       └── litellm_svc.py# Обёртка LiteLLM, форматирование JSON-ответов и промптов
├── tasks/                # Асинхронные воркеры (aiojobs / arq)
│   ├── scheduler.py      # Планировщик регулярных проверок каналов
│   └── feedback_worker.py# Обработка лайков/дизлайиков на лету
├── utils/
│   ├── redis_client.py   # Инстанс Redis, обработка очередей и буферов
│   └── alerting.py      # Механизм алертов админу при крашах (FloodWait, API 429)
└── tests/                # Юнит-тесты на асинхронные запросы и парсеры
```
