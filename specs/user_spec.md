# USER SPECIFICATION: JobSeeker AI Bot (MVP) - V2

## 1. Бизнес-цель и контекст
Автоматизировать поиск вакансий для соискателей, минимизируя информационный шум в Telegram. Система должна агрегировать данные с `hh.ru` (через официальный API) и отраслевых TG-каналов, профилировать пользователей через естественный язык, а затем с помощью AI анализировать релевантность вакансий. Ключевая ценность — точечная доставка только подходящих предложений без спама, с возможностью подгонки фильтра под пользователя (Feedback Loop).

## 2. Функциональные требования
- [ ] **Агрегация данных:** Парсинг вакансий через официальный API `hh.ru` (с контролем лимитов) + инкрементальный захват новых сообщений из целевых отраслевых каналов в Telegram с помощью юзер-бота.
- [ ] **Профилирование пользователей:** Хранение персональных кабинетов в БД, где пользователи задают диапазоны поиска и требования на естественном языке.
- [ ] **Фильтрация дубликатов:** Первичная склейка дубликатов вакансий по техническим идентификаторам (URL/ID вакансии в API или хэшу текста поста TG).
- [ ] **AI-Маршрутизатор (Адаптивный промпт):** Анализ вакансий через LLM, сопоставление с профилем юзера. Поддержка Feedback Loop (оценки "+" / "-") для динамической корректировки весов фильтра без дообучения модели.
- [ ] **Умная рассылка:** Бот отправляет сообщения только при наличии релевантных находок. Уведомления буферизируются и группируются (batching) в Redis, чтобы отправляться пакетами, не спамя пользователя.
- [ ] **Администраторский доступ:** Просмотр логов, ручной override фильтров, управление состоянием парсеров и мгновенные алерты админу при критических сбоях (баны API, капча).

## 3. Технологический стек и ограничения
- **Язык/Окружение:** Python 3.11+ (запуск в изолированном окружении WSL2).
- **Telegram Bot API:** `aiogram` v3.x (современный асинхронный фреймворк).
- **Telegram Client API (Парсинг каналов):** `Telethon` или `Pyrogram` (для работы технического юзер-бота с защитой от FloodWait).
- **Парсинг и API:** Асинхронный клиент под официальный API `api.hh.ru` со строгим соблюдением Rate Limits (до 30 запросов/сек) и задержками.
- **AI-Слой:** `LiteLLM` для унификации вызовов локальных/облачных моделей.
- **Базы данных и ORM:** `PostgreSQL` + `SQLAlchemy 2.0` (Async) + `JSONB` для хранения гибких профилей и весов фильтров.
- **Очереди и Кэш:** `Redis` для хранения очередей задач (Celery/ARQ), состояний буферизации рассылок и кэша уникальных ID вакансий.

## 4. Пограничные сценарии (Edge Cases) и обработка ошибок
- **Сбои и лимиты API hh.ru:** При получении `429 Too Many Requests` система включает экспоненциальную задержку (паузу), временно замораживает парсер, отправляет алерт админу и продолжает работать с внутренним кэшем ранее загруженных данных.
- **Сбои и лимиты Telegram (FloodWait):** Юзер-бот обрабатывает ошибки `FloodWaitError`, автоматически выставляет случайные задержки (2–5 сек) между запросами к каналам и собирает данные инкрементально.
- **Временная недоступность LLM / Ошибки сети:** Если локальный или внешний сервис нейросети недоступен, вакансия не отбрасывается. Система помещает её в отказоустойчивую очередь в `Redis` для повторного анализа ИИ позже, как только сервис вернется в сеть.
- **Пустой результат фильтрации:** Если под специфический запрос пользователя за сутки ИИ не нашел совпадений, бот умолкает в течение дня, но раз в сутки отправляет вежливый отчет: *"За последние 24 часа новых вакансий по вашему фильтру не найдено, я продолжаю поиск"*.
- **Критический отвал инфраструктуры:** При падении БД или общей сети система переходит в режим Graceful Degradation: парсеры встают на паузу, активные сессии юзеров получают уведомление о технических работах, админу улетает лог ошибки.
